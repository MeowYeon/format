# Agent Guide

This is a personal Obsidian plugin named `Format`. It applies custom Markdown formatting rules through a manual command/context-menu action and a close-time preview flow.

Use this file as the first stop before scanning the repository. Prefer the module index below to locate the relevant files, then read only the files needed for the requested change.

## Project Overview

- Runtime: CommonJS JavaScript, bundled for Obsidian with esbuild.
- Plugin entry: `main.js`.
- Pure formatter logic: `formatter.js`.
- Tests: Node's built-in test runner with Markdown fixture files under `case/`.
- Release output: `release/format/`, generated by `npm run build:release`.
- Obsidian metadata: `manifest.json`.

## Common Commands

- Run all tests: `npm test`
- Build release bundle: `npm run build:release`

Run `npm test` after formatter behavior changes. Run `npm run build:release` after changes that affect the shipped plugin bundle or release packaging.

## Shell Execution

- This repository lives in WSL. Run project commands inside WSL, not directly from PowerShell against the `\\wsl$` path.
- Preferred shape from Windows-hosted Codex sessions: `wsl -d Debian --cd /home/ghp/space/ai/format <command>`.
- Examples: `wsl -d Debian --cd /home/ghp/space/ai/format npm test` and `wsl -d Debian --cd /home/ghp/space/ai/format npm run build:release`.

## Directory Navigation

- `main.js`: Obsidian plugin integration, command registration, editor context menu, file close detection, preview modal, and vault writes.
- `formatter.js`: Markdown parsing and text transformation rules. Keep this independent from Obsidian APIs.
- `formatter.test.js`: Test runner that loads every `case/case-*.md` fixture and checks `formatText`.
- `case/`: Markdown fixtures. Each file contains an input block and an expected output block for one formatting scenario.
- `styles.css`: Styles for the close-time formatting preview modal.
- `manifest.json`: Obsidian plugin id, display name, version, minimum app version, description, author, and platform flag.
- `versions.json`: Obsidian plugin version compatibility map.
- `scripts/build-release.js`: esbuild release script. Bundles `main.js` to `release/format/main.js` and keeps `obsidian` external.
- `release/`: Generated release folder. Do not edit generated bundle contents by hand.
- `node_modules/`: Installed dependencies. Ignore by default.
- `.codex`: Local Codex scratch/config placeholder. Ignore by default.

## Module Index

### Obsidian Integration

Relevant files:
- `main.js`
- `styles.css`
- `manifest.json`

Responsibilities:
- Register the `custom-format` command.
- Add the editor right-click menu item titled `Custom Format`.
- Format either the current selection or the full document.
- Track open Markdown files and detect file close/layout changes.
- Show a preview modal before applying close-time formatting.
- Modify vault files only after the user confirms.

Common entry points:
- Manual formatting behavior: `formatSelectionOrDocument` in `main.js`.
- Close-time formatting behavior: `handleLayoutChange`, `handleAppQuit`, and `maybeFormatOnClose` in `main.js`.
- Preview UI: `FormatPreviewModal` in `main.js` and `.format-preview-*` selectors in `styles.css`.

Notes:
- Keep Obsidian API usage inside `main.js`.
- The formatter should be called through `formatText`.
- Avoid writing to the vault unless the latest file content has been read and checked.

### Formatting Rules

Relevant files:
- `formatter.js`
- `formatter.test.js`
- `case/`
- `README.md`
- `explain.md`

Responsibilities:
- Convert `;;text;;` into inline backtick code spans.
- Convert `;;+` into a backtick span from that marker to the end of the line.
- Normalize heading marker spacing, such as `#Title` to `# Title`.
- Normalize ordered and unordered list marker spacing.
- Add two trailing spaces to non-empty list item lines.
- Leave fenced code blocks unchanged.
- Return both formatted output and a per-line change summary.

Common entry points:
- Main API: `formatText(input)` in `formatter.js`.
- AST-derived formatting plan: `buildFormattingPlan`.
- Shorthand rules: `applyLineShorthand` and `replaceInlineBacktickShorthand`.
- Heading rules: `normalizeHeadingLine` and `normalizeHeadingFallbackLine`.
- List rules: `normalizeListItemLine` and `normalizeListFallbackLine`.
- Change preview data: `summarizeChanges`.

Notes:
- `formatter.js` dynamically imports `unified`, `remark-parse`, and `unist-util-visit`.
- Prefer adding or updating Markdown fixtures in `case/` for formatter behavior changes.
- Keep formatter behavior deterministic and free of Obsidian state.
- Be careful with trailing spaces in expected outputs; list fixtures may intentionally end lines with two spaces.

### Test Fixtures

Relevant files:
- `formatter.test.js`
- `case/case-*.md`

Responsibilities:
- Cover formatting rules with input and expected output examples.
- Provide regression cases for interactions between rules.

Fixture format:
- The test runner reads files matching `case/case-*.md`.
- Each fixture must contain a title heading.
- Each fixture must contain a fenced input section and fenced expected-output section matching the section names already used by the existing fixtures.

Notes:
- Some section headings may appear garbled in PowerShell output because of encoding display, but the existing test runner matches the literal file contents.
- When adding a fixture, copy an existing `case/case-*.md` file and edit the fenced blocks to preserve section heading text exactly.

### Release Packaging

Relevant files:
- `scripts/build-release.js`
- `package.json`
- `manifest.json`
- `versions.json`
- `release/`

Responsibilities:
- Bundle plugin code with esbuild.
- Keep Obsidian as an external dependency.
- Emit release assets under `release/format/`.

Common entry points:
- Build command: `npm run build:release`.
- Build script: `scripts/build-release.js`.

Notes:
- Do not manually edit generated files in `release/` unless the task is explicitly about release artifacts.
- If plugin metadata changes, check whether `manifest.json`, `versions.json`, and package metadata should stay aligned.

## Reading Paths

### Change a Formatting Rule

1. Read the relevant function in `formatter.js`.
2. Read nearby fixtures in `case/`.
3. Add or update a fixture before or alongside the implementation change.
4. Run `npm test`.

### Change Manual Command or Context Menu Behavior

1. Read `formatSelectionOrDocument` in `main.js`.
2. Check command/menu registration in `onload`.
3. Run `npm test` if formatter output is affected.
4. Run `npm run build:release` if the plugin bundle should be refreshed.

### Change Close-Time Preview Behavior

1. Read `handleLayoutChange`, `handleAppQuit`, and `maybeFormatOnClose` in `main.js`.
2. Read `FormatPreviewModal` in `main.js`.
3. Read `styles.css` if layout or preview presentation changes.
4. Run `npm run build:release` after integration changes.

### Change Release Build

1. Read `scripts/build-release.js`.
2. Check `package.json` scripts and dependencies.
3. Run `npm run build:release`.

## Editing Guidelines

- Keep formatter logic separate from Obsidian integration.
- Keep changes small and direct; this is a personal plugin, so prefer clarity over framework-like abstraction.
- Use CommonJS style for local source files unless the project is intentionally migrated.
- Preserve `"use strict";` in JavaScript files.
- Do not rewrite generated release bundles as source changes.
- Do not edit `node_modules/`.
- Treat `.codex` as local-only and ignored.
- Preserve intentional trailing spaces in Markdown fixtures.
- Avoid broad repository scans once the relevant module is identified from this guide.
